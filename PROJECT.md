## 2. プロジェクト概要

### 2.1 目的

ユーザーのウィッシュリストを「今月やること」として可視化し、優しい通知で第一歩を促すモバイルアプリを提供する。
マイクロマネジメントではなく、ユーザーに寄り添いながら目標達成をサポートし、紙やノートでは実現できない「日常的に目標を意識する仕組み」を提供する。

### 2.2 サービス概要

- **カテゴリー管理**: HOME画面の最大単位。プリセット + 自由作成
- **ウィッシュリストアイテム**: やりたいことを登録（複数カテゴリーに所属可能）
- **ステップ**: AIが低ハードルな第一歩を提案、ユーザーがポチポチ選択
- **今月やること**: 最大10個を選択し、ホーム画面とウィジェットで常に意識
- **優しい通知**: ユーザーの反応を学習し、最適なタイミングで通知
- **SNSシェア**: 完了したウィッシュリストをオシャレな画像に変換してシェア

### 2.3 ターゲットユーザー

| ペルソナ | 説明 | 主な利用機能 |
|---------|------|-------------|
| 30代女性会社員 | 仕事とプライベートの両立を目指す、自己成長や旅行・趣味に関心が高い | ウィッシュリスト作成、AI提案、リマインド、SNSシェア |
| 20代男性学生 | 将来の目標やスキル習得に意欲的、SNSでの情報発信に慣れている | AI提案ステップ、SNSシェア |
| 40代主婦/主夫 | 家族の予定管理と自分の趣味を両立したい | カテゴリ管理、リマインド、ウィジェット |

## 3. 機能要件定義

### 3.1 デバイス認証・データ管理

#### 3.1.1 匿名利用（Device ID認証）

**概要**: ユーザーはアカウント登録なしで即座にアプリを利用開始できる。データはDevice IDと紐づけてサーバーに保存される。

**ユーザーストーリー**:
- 新規利用者として、面倒な登録なしですぐにアプリを使い始めたい。なぜなら登録の手間で離脱したくないから。

**受け入れ条件**:
- [ ] 初回起動時にDevice ID（UUID）を自動生成
- [ ] Device IDをサーバーに送信し、ユーザーデータと紐づけ
- [ ] Device IDはセキュアストレージに保存
- [ ] 利用規約・プライバシーポリシーへの同意チェックが必要（初回のみ）

**実装ステータス**: 未着手

#### 3.1.2 データ永続化

**概要**: Device IDベースでユーザーデータをサーバーに保存・同期する。

**ユーザーストーリー**:
- ユーザーとして、アプリを閉じてもデータが消えないようにしたい。なぜなら作成したウィッシュリストを失いたくないから。

**受け入れ条件**:
- [ ] ウィッシュリスト・ステップ・設定がDevice IDと紐づいて保存される
- [ ] アプリ起動時にサーバーからデータを取得
- [ ] オフライン時はローカルキャッシュを使用
- [ ] ネットワーク復帰時に自動同期

**実装ステータス**: 未着手

- [ ] 生成AIが文脈に応じて内容を作成（励まし系/質問系/進捗確認系）
- [ ] 同じ文面が連続しないようバリエーションを持たせる

**通知例**:
| パターン | 例 |
|----------|-----|
| ステップ確認 | 「旅行雑誌を買う、できましたか？」 |
| 週末提案 | 「今週末はヨーロッパ旅行のYouTubeを見てみましょう」 |
| 励まし | 「英語学習、小さな一歩から始めてみませんか？」 |

**実装ステータス**: 未着手

#### 3.7.3 スヌーズ機能

**概要**: 通知を開いたが今は取り組めない場合、次回の通知タイミングを選択できる。

**ユーザーストーリー**:
- ユーザーとして、今は忙しいので後で通知してほしい。なぜなら無視するのは気が引けるから。

**受け入れ条件**:
- [ ] スヌーズ期間を選択可能（1週間後/2週間後/3週間後/5週間後/8週間後）
- [ ] 選択した期間後に再度通知
- [ ] スヌーズ履歴もDBに記録

**実装ステータス**: 未着手

#### 3.7.4 通知タイミング自動調整（裏側ロジック）

**概要**: ユーザーの行動パターンから最適な通知タイミングを非同期バッチで算出する。

**ユーザーストーリー**:
- ユーザーとして、ちょうどいいタイミングで通知を受けたい。なぜなら忙しい時に通知が来ると邪魔だから。

**受け入れ条件**:
- [ ] 登録時間、ログイン時間、操作ログをDBに記録
- [ ] 非同期バッチでユーザーごと×通知カテゴリーごとの最適な曜日・時間を算出
- [ ] 算出結果をDBに保持し、通知送信時に参照
- [ ] 例：登録が23時のユーザーには翌日も23時頃に送信
- [ ] 例：平日日中の利用が多いユーザーには、平日は日中に送信

**実装ステータス**: 未着手

#### 3.7.5 通知頻度の自動調整

**概要**: ユーザーの反応に基づいて頻度を自動調整する。

**ユーザーストーリー**:
- ユーザーとして、反応しない通知は減らしてほしい。なぜなら煩わしいから。

**受け入れ条件**:
- [ ] 通知への反応を記録（開いた/無視/スヌーズ）
- [ ] 無視が続く場合は通知頻度を段階的に下げる
- [ ] ユーザーが明示的に通知オフにした場合は尊重

**実装ステータス**: 未着手

## 5. ビジネス制約

### 5.1 運用制約

- プッシュ通知は深夜帯（22:00-7:00）を避けてデフォルト設定
- サーバーメンテナンスは事前告知の上、深夜帯に実施

### 5.2 法規制・コンプライアンス

- 個人情報保護法の遵守
- GDPR対応（データ削除リクエスト、データポータビリティ）
- 利用規約・プライバシーポリシーへの明示的な同意取得
- 通知許可のOS標準ダイアログを使用

### 5.3 データプライバシー

- ユーザーのウィッシュリストは暗号化されたクラウド（AES-256）に保存
- 全API通信はHTTPS/TLSにて暗号化
- 第三者へのデータ提供はユーザー同意がなければ行わない

## 7. 今後の拡張性考慮事項

### Phase 2: 拡張機能

- **アカウント登録・クラウド同期**: メールアドレスまたはソーシャルログイン（Google/Apple）でアカウント作成。Device IDのデータをアカウントに紐づけ、複数端末間でのデータ同期を実現。機種変更時のデータ移行にも対応。

### Phase 3: 収益化

- **アフィリエイト連携**: AIが提案するアクション（旅行予約、オンライン講座など）にアフィリエイトリンクを挿入し、クリック・購入に応じて収益を得る。

### その他検討事項

- ウィッシュリストのテンプレート機能（人気の目標をワンタップで追加）
- 友人とのウィッシュリスト共有・応援機能
- カレンダーアプリとの連携
