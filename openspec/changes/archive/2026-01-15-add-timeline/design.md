# Design: タイムライン機能

## Context

ウィッシュリスト管理アプリに「タイムライン」機能を追加する。ユーザーは以下の2種類のエントリーを時系列で確認できる：

1. **ジャーナル**: ユーザーが自由に書く日々の気づき・振り返り
2. **完了ログ**: ウィッシュリストアイテムやステップが完了した際の自動記録

### 制約

- 既存の Device ベースの認証システムを使用
- Notion風ミニマルデザインを踏襲
- モバイルアプリ（Expo + React Native）での実装
- 既存の `WishlistItem`、`Step` テーブルの `completedAt` フィールドを活用

## Goals / Non-Goals

### Goals

- シンプルで気軽に書けるジャーナル入力体験
- ジャーナルと完了ログを統合した時系列表示
- HOME画面との自然なナビゲーション連携
- 目標達成の軌跡を振り返れる体験

### Non-Goals

- 複雑なタグ付けやカテゴリ分類（将来の拡張）
- 画像・メディア添付（将来の拡張）
- 検索機能（将来の拡張）
- AI による振り返り分析（将来の拡張）

## Decisions

### 1. ナビゲーション構成

**決定**: スワイプ可能なタブナビゲーション（タイムライン ← HOME → Settings）

**理由**:
- HOME が中心的な画面であることを維持
- 左スワイプでタイムライン、右スワイプで設定という直感的な操作
- ボトムナビゲーションバーは引き続き非表示（ミニマルデザイン）
- タブインジケーター（ドット）で現在位置を示す

**代替案**:
- ボトムタブバーを表示 → ミニマルデザインに反する
- ドロワーメニュー → 発見性が低い
- FABからモーダル → タイムライン専用タブの要件に合わない

### 2. タイムラインデータ構造

**決定**: ジャーナル専用テーブル + 既存の完了データを API 層で統合

```prisma
model JournalEntry {
  id        String    @id @default(cuid())
  deviceId  String
  title     String?   // 任意
  content   String    // 本文（最大5000文字）
  createdAt DateTime  @default(now())
  updatedAt DateTime  @updatedAt

  device Device @relation(fields: [deviceId], references: [id], onDelete: Cascade)

  @@index([deviceId])
  @@index([deviceId, createdAt])
}
```

**理由**:
- ジャーナルは新規テーブルで管理
- 完了ログは既存の `WishlistItem.completedAt`、`Step.completedAt` を活用
- API の `getTimeline` で両方を取得し、クライアント側で時系列マージ
- 新規テーブルを最小限に抑えつつ、既存データを有効活用

**代替案**:
- 統一 TimelineEntry テーブル → 完了データの重複保存が必要
- 完了時にイベントテーブルに書き込み → 既存ロジックへの変更が大きい

### 3. タイムラインエントリーの種類

**決定**: 3種類のエントリータイプを定義

| タイプ | ソース | 表示内容 |
|--------|--------|----------|
| `journal` | JournalEntry テーブル | ユーザーが書いたタイトル・本文 |
| `item_completed` | WishlistItem.completedAt | 「🎉 [アイテム名] を達成しました」 |
| `step_completed` | Step.completedAt | 「✅ [ステップ名] を完了しました」 |

**理由**:
- 種類ごとに異なるUIスタイル（色、アイコン）で視覚的に区別
- ジャーナルは編集・削除可能、完了ログは読み取り専用

### 4. 入力UI

**決定**: タイムライン画面上部にフローティング入力ボタン（FAB）

**理由**:
- タイムラインは主に閲覧画面、入力は明示的なアクションとして分離
- FABタップでシートモーダルが開き、ジャーナルを入力
- 空状態では「今日の気づきを書いてみましょう」のプロンプトで誘導

**代替案**:
- インラインエディター → 一覧との混在で UI が複雑化
- 別画面での入力 → 画面遷移が煩雑

### 5. 一覧表示

**決定**: 日付でグループ化した逆時系列リスト

**理由**:
- 最新のエントリーが上に表示され、振り返りが容易
- 日付ヘッダー（例：「今日」「昨日」「1月15日」）で時系列を把握
- ジャーナルと完了ログが混在しても、日付グループで整理される

### 6. HOME からの導線

**決定**: ヘッダー左側にタイムラインアイコンを配置

**理由**:
- タイムラインタブが左にあることと一貫性がある
- タップでタイムラインタブにスイッチ

## Risks / Trade-offs

| リスク | 影響 | 緩和策 |
|--------|------|--------|
| スワイプナビの実装複雑性 | 中 | react-native-pager-view 等の実績あるライブラリを使用 |
| 大量の完了ログによるパフォーマンス | 中 | ページネーション（50件ずつ）、日付範囲でのフィルタリング |
| 長文入力時のパフォーマンス | 低 | 5000文字制限、デバウンスによる保存 |
| オフライン時のデータロス | 中 | ローカルストレージでドラフト保存 |

## Migration Plan

1. DB マイグレーション（JournalEntry テーブル追加）
2. バックエンド API 追加（getTimeline, createJournalEntry 等）
3. モバイルアプリのナビゲーション変更
4. タイムライン画面実装

新規テーブルの追加のみで既存データへの影響なし。ロールバックは不要。

- ジャーナルエントリーにムード（気分）を記録するオプションを追加するか？（将来の拡張として検討）
